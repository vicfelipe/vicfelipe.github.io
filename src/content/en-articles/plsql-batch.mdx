## The Context

In a legacy agricultural ERP system, there was a nightly closing routine that processed the day's orders: calculated taxes, updated inventory, and generated accounting entries. As the business grew, this routine went from 20 minutes to over 4 hours — unacceptable for a nightly processing window.

The original code used a cursor with row-by-row `FETCH`, the classic (and slow) PL/SQL pattern.

## The Problem: Row-by-Row is Slow-by-Slow

Row-by-row processing in Oracle suffers from a phenomenon called "context switching": each SQL operation switches the execution context between the PL/SQL engine and the SQL engine. With 10 million records, that means 10 million context switches.

```sql
-- Original code: too slow
DECLARE
  CURSOR c_orders IS
    SELECT * FROM orders WHERE order_date = TRUNC(SYSDATE - 1);
  v_order orders%ROWTYPE;
BEGIN
  OPEN c_orders;
  LOOP
    FETCH c_orders INTO v_order;
    EXIT WHEN c_orders%NOTFOUND;

    UPDATE inventory
    SET quantity = quantity - v_order.qty
    WHERE product_id = v_order.product_id;

    INSERT INTO accounting_entries
    VALUES (v_order.order_id, v_order.amount, SYSDATE);
  END LOOP;
  CLOSE c_orders;
  COMMIT;
END;
```

## BULK COLLECT: Loading Into Memory

`BULK COLLECT` eliminates context switches by loading multiple rows at once into PL/SQL collections. The secret lies in the `LIMIT` parameter:

```sql
DECLARE
  TYPE t_orders IS TABLE OF orders%ROWTYPE;
  v_orders t_orders;
  CURSOR c_orders IS
    SELECT * FROM orders WHERE order_date = TRUNC(SYSDATE - 1);
  v_batch_size CONSTANT PLS_INTEGER := 1000;
BEGIN
  OPEN c_orders;
  LOOP
    FETCH c_orders BULK COLLECT INTO v_orders LIMIT v_batch_size;
    EXIT WHEN v_orders.COUNT = 0;

    process_batch(v_orders);
    COMMIT;
  END LOOP;
  CLOSE c_orders;
END;
```

`LIMIT 1000` is critical: without it, Oracle loads **all** records into memory at once — a potential OOM risk with large tables.

## FORALL: Bulk DML

`FORALL` is the complement of `BULK COLLECT` for DML operations. Instead of iterating over the collection with a loop, `FORALL` sends all operations to the SQL engine at once:

```sql
CREATE OR REPLACE PROCEDURE process_batch(p_orders IN t_orders) AS
  TYPE t_ids IS TABLE OF orders.order_id%TYPE;
  TYPE t_qtys IS TABLE OF orders.qty%TYPE;
  TYPE t_prods IS TABLE OF orders.product_id%TYPE;

  v_ids   t_ids   := t_ids();
  v_qtys  t_qtys  := t_qtys();
  v_prods t_prods := t_prods();
BEGIN
  v_ids.EXTEND(p_orders.COUNT);
  v_qtys.EXTEND(p_orders.COUNT);
  v_prods.EXTEND(p_orders.COUNT);

  FOR i IN 1..p_orders.COUNT LOOP
    v_ids(i)   := p_orders(i).order_id;
    v_qtys(i)  := p_orders(i).qty;
    v_prods(i) := p_orders(i).product_id;
  END LOOP;

  -- Single context switch for N updates
  FORALL i IN 1..v_prods.COUNT
    UPDATE inventory
    SET quantity = quantity - v_qtys(i)
    WHERE product_id = v_prods(i);

  -- Single context switch for N inserts
  FORALL i IN 1..v_ids.COUNT
    INSERT INTO accounting_entries
    SELECT v_ids(i), amount, SYSDATE
    FROM orders
    WHERE order_id = v_ids(i);

END process_batch;
```

## Error Handling with SAVE EXCEPTIONS

`FORALL` by default stops at the first error. To process as much as possible and log individual errors, use `SAVE EXCEPTIONS`:

```sql
FORALL i IN 1..v_prods.COUNT SAVE EXCEPTIONS
  UPDATE inventory
  SET quantity = quantity - v_qtys(i)
  WHERE product_id = v_prods(i);

EXCEPTION
  WHEN OTHERS THEN
    IF SQLCODE = -24381 THEN
      FOR i IN 1..SQL%BULK_EXCEPTIONS.COUNT LOOP
        log_error(
          SQL%BULK_EXCEPTIONS(i).ERROR_INDEX,
          SQL%BULK_EXCEPTIONS(i).ERROR_CODE
        );
      END LOOP;
    ELSE
      RAISE;
    END IF;
```

## Results

After refactoring to BULK COLLECT + FORALL with 1000-record batches:

| Metric | Before | After | Reduction |
|---|---|---|---|
| Total time | 4h 12min | 23min | **91%** |
| Context switches | ~10M | ~10k | **99.9%** |
| UNDO generated | 18 GB | 2.1 GB | **88%** |
| Concurrent locks | ~500 | ~15 | **97%** |

## Calibrating LIMIT

The ideal batch size depends on the environment:

- **Too small** (< 100): little reduction in context switches
- **Too large** (> 10000): OOM risk, long locks, excessive undo
- **Typical sweet spot**: 500–2000 for tables with rows < 500 bytes

Use Oracle AWR to monitor `PGA aggregate target` and adjust based on available memory.
